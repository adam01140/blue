1. PURPOSE AND INTENDED GOAL
================================================================================

FormStar is a product that turns government PDF forms into friendly HTML surveys,
fills the original PDF from user answers, and reuses personal data across forms.

The **Auto-Form Creator** is a shareholder/demo pipeline that automates the hardest
part of that workflow:

  Government XFA PDF  →  unlocked AcroForm PDF  →  semantic field names
       →  AI-generated field mapping  →  AI-generated survey config
       →  HTML form with autofill  →  filled PDF preview

**Intended goal (production vision):**
  - Upload any government PDF form
  - Automatically produce a FormStar-quality HTML form wired to the PDF
  - User profile data (name, address, DOB, etc.) autofills across forms
  - Submitting the HTML form fills the source PDF correctly

**Current scope (demo):**
  - Interactive walkthrough at demo.html (Steps 1–8)
  - Uses OpenAI (gpt-4o-mini) for Steps 4 and 6
  - Uses Firebase Auth + Firestore for user profile
  - Test form: public/Auto-Form-Creator/xfa-test.pdf (BCIA 8016-style; ~43 text + 5 checkbox fields after prepare)

This demo lives inside the larger FormStar repo (server.js, public/Forms/*, etc.).
Production forms like SC-100 use the same PDF-filling model documented in
pdf-filling-documentation.txt at the repo root.

2. HIGH-LEVEL ARCHITECTURE
================================================================================

  Browser (demo.html)
      │
      ├── Firebase Auth ──► Firestore users/{uid}  (user profile JSON)
      │
      ├── pdf.js (client) ──► extract fields from PDF
      │
      └── fetch() ──► Node server (server.js, port 3000)
                          │
                          ├── unlock-pdf-handler.js      (XFA → AcroForm)
                          ├── pdf-field-deduplicator.js  (radio → unique checkboxes)
                          ├── pdf-field-sanitizer.js     (rename fields, flatten hierarchy)
                          ├── field-config-generator.js  (OpenAI → field_config.json)
                          ├── form-config-generator.js   (OpenAI → form_config.json)
                          ├── form-autopopulate.js       (profile ↔ form config enrichment)
                          ├── auto-form-pdf-handler.js   (store/fill sanitized PDF)
                          ├── pdf-form-filler.js         (pdf-lib fill logic)
                          └── openai-fetch.js            (OpenAI HTTP with IPv4 + retries)

  Client-side HTML generation:
      form-html-renderer.js builds standalone HTML from form_config.json + profile


Critical invariant (same as production FormStar):
  HTML input `name` attribute  ===  PDF field name after sanitization (field.newName)
  This is how /edit_pdf and /api/auto-form/fill-pdf know which widget to fill.


================================================================================
3. THE 8-STEP DEMO PIPELINE
================================================================================

Step 1 — Sign In
  - Firebase email/password auth
  - Loads Firestore document users/{uid} into lastUserProfile
  - Shows profile JSON + editable "User Profile Entries" grid (2 columns)
  - Entries can be added, edited (key locked), or deleted (uid is not deletable)

Step 2 — Unlock PDF
  - Upload XFA PDF
  - POST /api/unlock-pdf strips XFA, produces AcroForm PDF
  - Download unlocked PDF

Step 3 — Extract Fields
  - Upload prepared/unlocked PDF
  - Client pdf-extractor.js extracts:
      [[field_name]] for text fields
      {{field_name}} for checkboxes
  - Also extracts document layout text for AI context
  - POST /api/prepare-pdf-fields deduplicates radio groups into unique checkboxes

Step 4 — Generate field_config.json
  - ChatGPT via POST /api/generate-field-config (~60s progress bar)
  - Prompt: public/Auto-Form-Creator/field_config.txt
  - Output maps every PDF field id → semantic newName, type, label
  - Example: form1[0].#subform[0].city[0] → applicant_home_city

Step 5 — Sanitize PDF
  - POST /api/sanitize-pdf renames PDF fields to semantic newName values
  - Promotes fields to top-level AcroForm (removes form1[0] hierarchy)
  - Fixes checkbox widget Parent/Annots issues (MorF sex checkboxes must survive)
  - POST /api/auto-form/store-pdf registers sanitized PDF server-side → pdfToken
  - pdfToken is stored in lastFormConfig.pdfToken for HTML generation

Step 6 — Generate form_config.json
  - ChatGPT via POST /api/generate-form-config (~120s progress bar)
  - Prompt: public/Auto-Form-Creator/form_config.txt
  - Output: sections/questions with nameId, type, autopopulate flags
  - Options:
      Form layout: all_at_once | one_at_a_time
      HTML output mode: dev | normal  (stored as htmlMode on form_config)
  - Retry + injectMissingFieldQuestions fallback if ChatGPT omits fields
  - enrichFormConfigAutopopulate() adds autopopulateAvailable/Missing arrays

Step 7 — User Info Fields (autopopulate analysis)
  - Shows which fields can autofill from profile vs missing
  - POST /api/enrich-autopopulate refreshes after profile edits
  - Yellow "Not in profile yet" pills; green "Available" pills

Step 8 — Generate Form HTML
  - FormHtmlRenderer.build(formConfig, userProfile, options)
  - Buttons: Generate Form HTML | View HTML | Download HTML
  - View HTML toggles inline HTML source (with copy icon on code toolbar)
  - Live preview in iframe (same-origin document.write, not blob URL)
  - Generated HTML includes Submit → filled PDF preview

4. DATA ARTIFACTS AND HOW THEY CONNECT
================================================================================

field_config.json
  {
    "formTitle": "...",
    "fields": [
      { "id": "raw_pdf_name", "newName": "applicant_first_name", "type": "text", "label": "..." }
    ]
  }
  - id = exact PDF name before sanitize
  - newName = semantic name used after Step 5 and in HTML name attributes

form_config.json
  {
    "formTitle": "...",
    "displayMode": "all_at_once" | "one_at_a_time",
    "htmlMode": "dev" | "normal",
    "pdfToken": "uuid-from-store-pdf",
    "sections": [{
      "sectionName": "...",
      "questions": [{
        "text": "What is your city?",
        "type": "text",
        "nameId": "applicant_home_city",   ← must match field_config newName
        "autopopulate": {
          "enabled": true,
          "profileKey": "city",
          "profileDescription": "User city"
        }
      }]
    }],
    "autopopulateAvailable": [...],
    "autopopulateMissing": [...]
  }

Sanitized PDF
  - Binary PDF with AcroForm field names = newName values
  - Registered on server with pdfToken (in-memory, 24h TTL)

Generated HTML
  - Standalone document with embedded Firebase, profile JSON snapshot, pdfToken
  - Input name attributes = question.nameId (or checkbox option nameId)


Pipeline dependency chain:
  Step 3 extraction → Step 4 field_config → Step 5 sanitized PDF + pdfToken
  → Step 6 form_config → Step 8 HTML


================================================================================
5. AUTOFILL / USER PROFILE SYSTEM
================================================================================

Storage:
  Firestore collection: users
  Document ID: Firebase uid
  Flat key-value profile, e.g.:
    firstName, lastName, city, street, zip, phone, email, ...
  Custom entries also store {key}_description for human-readable labels

Profile UI (demo.html Step 1):
  - Shows all keys except *_description companion keys
  - 2-column card grid with Edit/Delete buttons
  - uid entry has no delete/edit

How autofill is decided (form-autopopulate.js):
  1. ChatGPT sets autopopulate.enabled per question in form_config
  2. ensureAutopopulateOnFormConfig() enriches/heuristically fixes flags
  3. isLikelyAutopopulatableField() catches hair, eye, height, DL, address, etc.
  4. NON_AUTOPOPULATE_PATTERNS excludes agency fields (ori_code, billing_number, etc.)
  5. PROFILE_KEY_ALIASES maps profileKey ↔ existing Firestore keys (city, firstName, etc.)

Step 7 display:
  - autopopulateAvailable: field exists in profile with value
  - autopopulateMissing: autopopulatable but not yet in profile

Generated HTML autofill (form-html-renderer.js):
  - Embeds userProfile JSON at build time
  - Firebase Auth runs silently (normal mode) or visibly (dev mode)
  - onAuthStateChanged loads fresh Firestore profile and applies to inputs
  - Inputs with data-autopopulate + data-profile-key save on blur/change
  - postMessage to parent demo: { type: 'formstar-profile-update', key, value, description }
  - Dev mode badges: "Autofill available" / "Will save to profile"
  - Typing in a field updates badge to "Autofill available" live
  - Normal mode (htmlMode=normal): hides auth bar, badges, question numbers, toasts

Standalone autofill requirement:
  - HTML must be served over HTTP (localhost:3000), not file://
  - Firebase auth persists in localStorage on same origin
  - iframe-test.html and form-preview-shell.html use same-origin iframe document.write


================================================================================
6. GENERATED HTML FORMS
================================================================================

Builder: public/Auto-Form-Creator/form-html-renderer.js
  FormHtmlRenderer.build(formConfig, userProfile, options)
  options:
    firebaseConfig  — embedded for standalone auth
    htmlMode        — 'dev' | 'normal'
    pdfToken        — for PDF submit

Question types supported:
  text, bigParagraph, date, phone, money, dropdown, checkbox, multipleTextboxes

displayMode:
  all_at_once     — all questions visible
  one_at_a_time   — stepper with Previous/Next

htmlMode:
  dev (default)   — sign-in panel, autofill badges, numbered stepper text, toasts
  normal          — clean production UI; Firebase still runs in background

Submit section:
  - Submit button at end of form
  - POST /api/auto-form/fill-pdf/{pdfToken} with FormData
  - Checkboxes send 'on' when checked (matches shouldCheck in pdf-form-filler.js)
  - Dates converted yyyy-mm-dd → mm/dd/yyyy
  - Response PDF shown in iframe below button


================================================================================
7. PDF SUBMIT AND FILLED-PDF PREVIEW
================================================================================

Auto-Form uses a separate path from production /edit_pdf:

  POST /api/auto-form/store-pdf
    Body: multipart, field name "pdf" = sanitized PDF file
    Returns: { pdfToken, fileName }

  POST /api/auto-form/fill-pdf/:pdfToken
    Body: form field names → values (same as HTML input names)
    Returns: filled PDF bytes (application/pdf)

  pdf-form-filler.js — shared fill logic (pdf-lib):
    PDFTextField  → setText
    PDFCheckBox   → check/uncheck via shouldCheck (on, true, yes, 1, checked)
    PDFRadioGroup → mapRadioValue
    PDFDropdown   → select

pdfToken storage:
  - In-memory Map in auto-form-pdf-handler.js
  - TTL: 24 hours
  - Lost on server restart → user must re-run Step 5

Production /edit_pdf (server.js) still used by public/Forms/*.html:
  - Looks up PDF in public/Forms/ by filename
  - Same field-name matching rules apply

9. SERVER API ROUTES (AUTO-FORM SPECIFIC)
================================================================================

POST /api/unlock-pdf
  Body: pdf file → unlocked PDF bytes

POST /api/prepare-pdf-fields
  Body: pdf file → deduplicated PDF + field lists

POST /api/generate-field-config
  Body: { extractedDocumentContent, textFields, checkboxFields }
  Returns: { fieldConfig }

POST /api/sanitize-pdf
  Body: pdf file + fieldConfig JSON
  Returns: sanitized PDF bytes, X-Renamed-Count header

POST /api/generate-form-config
  Body: { fieldConfig, extractedDocumentContent, displayMode, userProfile }
  Returns: { formConfig }

POST /api/enrich-autopopulate
  Body: { formConfig, fieldConfig, userProfile, displayMode }
  Returns: enriched formConfig with available/missing arrays

POST /api/auto-form/store-pdf
  Body: pdf file → { pdfToken }

POST /api/auto-form/fill-pdf/:pdfToken
  Body: form fields → filled PDF bytes

POST /edit_pdf
  Production PDF fill (public/Forms/*.pdf by filename)

12. KNOWN ISSUES AND WORKAROUNDS
================================================================================

Firestore ERR_BLOCKED_BY_CLIENT
  - Ad blocker blocking firestore.googleapis.com
  - Fix: disable ad blocker for localhost or allow Firebase domains
  - Not an app bug

OpenAI DNS / fetch failed (ENOTFOUND api.openai.com)
  - Intermittent on Windows/corporate networks
  - Fixed via openai-fetch.js: explicit IPv4 HTTPS + retries
  - server.js sets dns.setDefaultResultOrder('ipv4first')
  - If still failing: check VPN/firewall/internet

pdf.worker.min.js: Warning TT: undefined function: 21
  - Harmless PDF.js font warning from certain PDFs; ignore

pdfToken lost after server restart
  - Re-run Step 5 (sanitize) to re-register PDF
  - Regenerate HTML in Step 8

Downloaded HTML opened as file://
  - Firebase auth will NOT work on file://
  - Serve over http://localhost:3000 or deploy to authorized Firebase domain

ChatGPT occasionally omits fields in form_config
  - form-config-generator.js retries once, then injectMissingFieldQuestions()

Adobe vs pdf.js field name mismatch (historical)
  - Fixed in pdf-field-sanitizer.js by promoting to top-level AcroForm
  - Must re-run Step 5 after sanitizer fixes

Step 4/6 take ~1–2 minutes
  - Progress bars are time-based estimates, not tied to actual API completion

13. DESIGN DECISIONS AND GOTCHAS
================================================================================

Why deduplicate radio groups to checkboxes?
  Government PDFs often have one radio group name (MorF[0]) with multiple widgets.
  HTML forms and /edit_pdf need unique name per widget. Deduplicator splits them
  to applicant_sex_male, applicant_sex_female, etc.

Why sanitize PDF field names?
  Raw names like form1[0].#subform[0].LastName[0] are unusable in HTML.
  Sanitizer renames to semantic snake_case matching field_config.newName.

Why pdfToken instead of embedding PDF in HTML?
  PDFs are large; in-memory server store keeps HTML portable while enabling fill.
  Token embedded in generated HTML and form_config.json.

Why two config files (field_config + form_config)?
  field_config = 1:1 PDF field mapping (structural)
  form_config = user-facing survey (questions, sections, autopopulate, layout)
  Separation lets AI focus each task and allows re-generation independently.

nameId must match exactly after sanitize:
  If form_config says nameId "city" but sanitizer renamed to "applicant_home_city",
  PDF fill silently skips the field. Validation in form-config-generator checks this.

Autopopulate profile keys:
  Prefer stable keys reused across forms (firstName, city, zip).
  ChatGPT assigns profileKey; heuristics and aliases reconcile with Firestore.

htmlMode dev vs normal:
  Dev = internal testing/demos with visible autofill machinery
  Normal = what end users would see in production
